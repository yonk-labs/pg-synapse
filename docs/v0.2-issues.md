# v0.2 SQL surface issues (deferred per N2.2)

N2.2 freezes the `synapse.*` SQL surface for v0.1.1. Kernel capabilities
land now; their SQL entry points are filed here and built in v0.2. This
is the durable record MASTER-PLAN refers to with "tracked inside
PS-1/4/7".

## V2-SQL-1: `synapse.execution_export(execution_id [, trace_level])`

- **Filed by:** PS-4 (redacted diagnostics export bundle).
- **Kernel capability shipped:** `pg_synapse_core::types::export_diagnostics`
  + `DiagnosticsBundle` (`crates/pg-synapse-core/src/types/diagnostics.rs`).
  Applies the PS-3 trace-level redaction discipline
  (`TraceLevel::should_persist_messages` / `should_persist_events`).
- **Deferred SQL:** a `synapse.execution_export(uuid)` SET-returning or
  JSON function that reads a finished run's persisted rows and returns
  the redacted bundle. Frozen until v0.2 (N2.2).
- **v0.2 design notes:** redact at the agent's stored `trace_level`
  (resolve like `resolve_trace_level` in `sql_functions.rs`); reuse the
  same `should_persist_*` gates so SQL output and the kernel bundle
  never disagree; no new tables (reads `synapse.executions` /
  `synapse.messages` / `synapse.traces`); grant to the same role set as
  the other read paths.

## V2-SQL-2: `synapse.provider_verify(...)`
